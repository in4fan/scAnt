# Przegląd kodu scAnt — 2025-07-02

## Podsumowanie

Projekt jest w dobrej kondycji architektonicznej — podział na API, CLI, worker i handler serverless jest czytelny. Poniżej zebrane błędy, potencjalne problemy i ścieżki rozwoju.

---

## 1. Błędy i problemy krytyczne

### 1.1 Konflikt nazwy modułu `watchdog`

W `api_server.py`:
```python
import watchdog
```
Moduł projektu nazywa się tak samo jak popularna biblioteka PyPI (`watchdog`). Jeśli kiedykolwiek `watchdog` (paczka) zostanie zainstalowana w środowisku (np. przez zależność tranzytywną), import się złamie lub pociągnie niewłaściwy moduł. 

**Rekomendacja:** Zmienić nazwę pliku na `health_monitor.py` lub `scant_watchdog.py`.

### 1.2 `asyncio.create_task` w kontekście shutdown (lifespan)

```python
yield
...
asyncio.create_task(watchdog.stop_watchdogs())
```

Po `yield` w lifespan event loop jest w fazie shutdown. `create_task` może nigdy nie zostać wykonany lub rzucić `RuntimeError` jeśli loop jest zamykany. Należy zamiast tego użyć `await`:

```python
await watchdog.stop_watchdogs()
```

### 1.3 Rate limiter nie działa z async endpointami

`rate_limit` dekorator jest synchroniczny (`def wrapper`), ale jest nakładany na synchroniczne endpointy FastAPI — to akurat działa. Problem jest inny: dekorator zwraca `wrapper` bez sygnatury oryginału, co psuje OpenAPI schema (FastAPI widzi `*args, **kwargs` zamiast prawdziwych parametrów).

**Rekomendacja:** Użyć `fastapi.Depends` z klasą limiter albo dedykowanej biblioteki (`slowapi`).

### 1.4 Niespójność zakresów pozycji Z pomiędzy Klipper config a kodem Python

- `skr_pico_klipper.cfg`: `position_min: -45000`, `position_max: 0`
- `Scanner_Controller.py`: `stepper_minPos = [0, -160, -450]`, domyślny scan range Z: `-250` do `-80`
- `ScanConfig` (pydantic): domyślne `z_min=-250`, `z_max=-80`

Klipper cfg pozwala na ruch do -45000, ale Python ogranicza do -450. Dokumentacja w kodzie mówi "mm/stopnie", ale -45000 mm to 45 metrów — coś jest nie tak z `rotation_distance` albo z limitem w cfg. **Potencjalne ryzyko kolizji mechanicznej** jeśli wartości się rozjadą.

**Rekomendacja:** Dodać walidację granic z wartości odczytanych z Moonrakera (`/printer/objects/query?configfile`) zamiast hardkodowanych.

### 1.5 `capture_single_image` zapisuje plik w CWD kontenera

```python
full_path = str(Path.cwd() / output_path)
```

W kontenerze `cwd` to `/app`. Użytkownik API może podać `output_path=../../etc/cron.d/evil` — to klasyczny **path traversal**. Brak sanitizacji.

**Rekomendacja:** Wymusić zapis wyłącznie wewnątrz `SCANS_DIR`:
```python
safe_path = Path(SCANS_DIR) / Path(output_path).name
```

### 1.6 `start_preview()` powoduje podwójny `start()`

`CameraController.__init__()` już wywołuje `self.picam2.start()`. Wywołanie `start_preview()` ponownie woła `start()` — picamera2 może rzucić wyjątek lub zachować się nieokreślenie przy podwójnym starcie.

## 2. Problemy średniego ryzyka

### 2.1 Brak `await` w `stop_watchdogs` — ponowne wskazanie

Funkcja `stop_watchdogs` jest `async def`, ale w lifespan jest wołana przez `asyncio.create_task(...)` bez `await`. Wynik:
- Taski mogą nie zdążyć się anulować przed zamknięciem serwera.

### 2.2 Brak timeout/retry na skanowaniu — nieskończona blokada

Jeśli Moonraker nie odpowie na `M400` (lub router API Moonrakera przestanie żyć), `send_gcode` rzuci wyjątek, ale `wait_for_moves` woła `send_gcode("M400")` — timeout 5s z requests i tyle. To OK, ale brak logiki reconnect. Jeśli `requests.post` z timeout 5s się powiedzie ale Klipper się zawiesi (M400 nigdy nie odpowie — Moonraker odeśle 200, bo to fire-and-forget), ruch nigdy się nie skończy a `time.sleep(settle_time)` i tak się wykona.

**Rekomendacja:** Po M400 poll `printer/objects/query?toolhead` i sprawdź flagę `homing` / ruch zakończony.

### 2.3 Thread-safety w `_ensure_hardware()`

`_ensure_hardware()` trzyma `_hardware_lock` (threading.Lock) ale `ScannerController` i `CameraController` nie są thread-safe — np. `send_gcode` może być wołany z tła (scan task) i jednocześnie z HTTP request (`/motor/move`) mimo checku `scanning_in_progress()`. Check i wykonanie nie są atomowe — race condition window jest mały, ale realny.

### 2.4 `processStack.py` — `data.split(" ")[1]` zakłada brak spacji w ścieżce

Jeśli ścieżka do obrazka zawiera spację, cały algorytm grupowania się rozsypie. Dotyczy to każdego użycia `data.split(" ")`.

### 2.5 `docker-compose.yml` — brak health check

Nie ma `healthcheck` w docker-compose, więc `restart: unless-stopped` nie wykryje zamrożenia procesu (deadlock bez crash).

**Rekomendacja:**
```yaml
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
  interval: 30s
  timeout: 5s
  retries: 3
```

### 2.6 `.gitignore` nieaktualny

Obecny `.gitignore` ignoruje `GUI/.idea/*` i `GUI/__pycache__/*` — te katalogi nie istnieją w projekcie (GUI zostało usunięte). Brakuje za to:
- `scans/`
- `*.egg-info/`
- `.venv/`
- `__pycache__/` (globalnie, nie tylko w podfolderach)

### 2.7 `requirements.txt` — `picamera2` na PC się nie zainstaluje

`picamera2` jest w `requirements.txt` ale wymaga RPi-specific libs. Docker build na PC (np. CI) się nie powiedzie.

**Rekomendacja:** Przenieść `picamera2` do osobnej grupy lub zainstalować ją warunkowo w Dockerfile tylko gdy target to ARM.

## 3. Drobne uwagi (code quality)

| Plik | Uwaga |
|------|-------|
| `Scanner_Controller.py` | `current_y = posY + (self.completedRotations * self.stepper_maxPos[1]) % 360` — operator `%` ma wyższy priorytet niż `+`. Prawdopodobnie zamierzony jest `(completedRotations * max) % 360`, ale warto dodać nawiasy dla jasności. |
| `processStack.py` | `data.split("\\")[-1]` — na Linuxie separator to `/`, więc ten split nigdy nie podzieli ścieżki. Użyć `Path(data).name`. |
| `handler.py` | Po `shutil.rmtree` w `finally` nie ma check czy upload się udał — jeśli upload padnie, dane są tracone bezpowrotnie. |
| `write_meta_data.py` | `get_default_values()` zawiera dane z kamery FLIR, nie RPI-HQ-CAMERA. Dead code / legacy. |
| `test_api.py` | Fixture monkeypatchuje `start_watchdogs` ale nie `stop_watchdogs` — przy cleanup może polecieć warning. |
| `api_server.py` | `app.mount(...)` wewnątrz `lifespan` jest niestandardowe — mount powinien być przed startem. Działa, ale warto przenieść mount'y przed `yield`. |
| `camera_controller.py` | `import cv2` z try/except + `cv2 = None` — potem `cv2.putText(...)` w `capture_image` jest wołany wewnątrz `if cv2 is not None` ale w `capture_jpeg_frame` warunek jest `if self.picam2 is None or cv2 is None` — logika jest poprawna, ale nieintuicyjna. |

## 5. Ścieżki rozwoju

### 5.1 WebSocket zamiast pollingu pozycji
Obecne GUI odpytuje `/motor/position` co 2s. WebSocket (lub SSE) dałby real-time feedback bez obciążania Moonrakera.

### 5.2 Konfiguracja kamery z poziomu API/GUI
Brak endpointu do zmiany parametrów ekspozycji (ISO, czas naświetlania). `CameraController.configure_exposure()` istnieje, ale nie jest wystawiony przez API.

### 5.3 Automatyczne wznawianie skanowania (resume)
Jeśli skanowanie się przerwie (awaria zasilania), nie ma mechanizmu wznowienia. Wystarczyłoby zapisywać checkpoint (ostatni zrobiony obraz) i pominąć już istniejące pliki.

### 5.4 Preview z autofokusem (focus peaking)
Strumień MJPEG mógłby nakładać focus peaking overlay (Laplacian colormap) — przydatne przy ręcznym ustawianiu ostrości.

### 5.5 Integracja z Meshroom / OpenMVS
Po stakkowaniu i maskowaniu naturalnym krokiem jest rekonstrukcja 3D. Pipeline mógłby automatycznie triggerować Meshroom na workerze.

### 5.6 Notification po zakończeniu skanowania
Webhook, e-mail lub push notification gdy długi scan się zakończy (user nie musi czekać przy terminalu).

### 5.7 Sterowanie oświetleniem (LED ring)
Dodanie sterowania LED ring przez GPIO Klippera (`SET_PIN`) — umożliwi programowalne oświetlenie sceny (np. różne kąty, intensywności) bez ręcznej ingerencji.

### 5.8 CI/CD pipeline
Brak jakiejkolwiek automatyzacji CI. Warto dodać GitHub Actions z:
- Lintem (`ruff`)
- Testami (`pytest`)
- Budową obrazów Docker (ARM + x86 multi-arch)

### 5.9 Pyproject.toml zamiast requirements.txt
Migracja na `pyproject.toml` z opcjonalnymi grupami (`[project.optional-dependencies]`) pozwoli rozwiązać problem z `picamera2` na PC (punkt 2.7).
