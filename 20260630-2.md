### 2. Globalny stan `_is_scanning` nie jest thread-safe w pełni
**Miejsce:** [api_server.py](api_server.py#L116), [api_server.py](api_server.py#L126), [api_server.py](api_server.py#L134)

**Problem:**
- `_is_scanning` jest zmienną globalną (linia 116)
- Mimo użycia `_scan_lock` (threading.Lock), globalny stan w Pythonie może powodować problemy w środowiskach wieloprocesowych (np. przy użyciu gunicorn z wieloma workerami)
- FastAPI z uvicorn w trybie default działa w jednym procesie, ale to założenie może się zmienić

**Skutek:**
- Race conditions w środowiskach wieloprocesowych
- Trudny do debugowania stan aplikacji

**Sugerowana naprawa:**
Przenieść stan do `app.state` i używać go jako single source of truth:
```python
# W lifespan:
app.state.is_scanning = False
app.state.scan_lock = threading.Lock()

# Zamiast global _is_scanning:
def scanning_in_progress():
    with app.state.scan_lock:
        return app.state.is_scanning
```

### 5. `getThreads()` w `processStack.py` nie jest cross-platform i Docker-safe
**Miejsce:** [processStack.py](processStack.py#L45), [processStack.py](processStack.py#L50)

**Problem:**
- Na Linux używa `grep -c cores /proc/cpuinfo` (linia 50)
- Na Windows używa `NUMBER_OF_PROCESSORS` (linia 48)
- W kontenerze Docker `/proc/cpuinfo` może być ograniczone (cgroups)
- `os.popen` jest podatne na injection i nie jest zalecane

**Skutek:**
- Niepoprawna liczba wątków w kontenerach Docker
- Potencjalne problemy z wydajnością (zbyt dużo/mało wątków)

**Sugerowana naprawa:**
```python
def getThreads():
    """Returns the number of available threads in a cross-platform, Docker-safe way."""
    try:
        # Najpierw spróbuj użyć os.cpu_count() (Python 3.4+)
        return os.cpu_count() or 1
    except Exception:
        # Fallback dla starszych Pythona
        if sys.platform == 'win32':
            return int(os.environ.get('NUMBER_OF_PROCESSORS', 1))
        else:
            # W kontenerach Docker /proc/cpuinfo może nie być dostępny
            # Użyj cgroups jeśli dostępny
            try:
                with open('/sys/fs/cgroup/cpu.max', 'r') as f:
                    # Format: "max period period_usec usage_usec"
                    parts = f.read().strip().split()
                    if len(parts) >= 2 and parts[0].isdigit():
                        return int(parts[0])
            except Exception:
                pass
            return 1
```

### 6. `parse_bool_arg` nie obsługuje `None` i niekonwencjonalnych wartości
**Miejsce:** [processStack.py](processStack.py#L33), [processStack.py](processStack.py#L37)

**Problem:**
- Jeśli `value` jest `None`, `str(None).strip().lower()` zwróci `"none"`
- `"none"` nie jest w żadnym z dopuszczalnych zestawów (`{"1", "true", ...}` ani `{"0", "false", ...}`)
- Zostanie rzucony `ArgumentTypeError`, co przerwie wykonywanie skryptu

**Skutek:**
- Skrypt może się wywalić przy opcjach CLI, które nie zostały podane

**Sugerowana naprawa:**
```python
def parse_bool_arg(value):
    if value is None:
        return False  # lub True, w zależności od domyślnego zachowania
    if isinstance(value, bool):
        return value
    
    normalized = str(value).strip().lower()
    if normalized in {"1", "true", "t", "yes", "y", "on", ""}:
        return True
    if normalized in {"0", "false", "f", "no", "n", "off", "none"}:
        return False
    
    raise argparse.ArgumentTypeError(f"Nieprawidłowa wartość bool: {value}")
```

### 9. `fetch_data` przerywa pobieranie przy błędzie pojedynczego pliku
**Miejsce:** [scant_cli.py](scant_cli.py#L84)

**Problem:**
- `raise_for_status()` (linia 85) w `download_file()` powoduje, że cały proces pobierania zostaje przerwany przy błędzie jednego pliku
- Brak mechanizmu retry dla pojedynczych plików

**Skutek:**
- Częściowe pobieranie projektów
- Konieczność ponownego uruchamiania pobierania od początku

**Sugerowana naprawa:**
```python
def download_file(f):
    url = f"{api}/scans_data/{project}/{f}"
    local_path = os.path.join(out_dir, f)
    os.makedirs(os.path.dirname(local_path), exist_ok=True)
    
    if os.path.exists(local_path):
        return True
    
    for attempt in range(3):  # 3 próby
        try:
            r = requests.get(url, stream=True, timeout=30)
            r.raise_for_status()
            with open(local_path, "wb") as out:
                for chunk in r.iter_content(chunk_size=8192):
                    if chunk:
                        out.write(chunk)
            return True
        except requests.exceptions.RequestException as e:
            logging.warning(f"Próba {attempt + 1}/3 nieudana dla {f}: {e}")
            if attempt == 2:  # Ostatnia próba
                logging.error(f"Ostateczny błąd pobierania {f}: {e}")
                return False
            time.sleep(2 ** attempt)  # Exponential backoff
    return False
```

### 13. Brak wykrywania zmian IP/MAC (wg AGENTS.md)
**Miejsce:** [watchdog.py](watchdog.py)

**Problem:**
- AGENTS.md wymaga: "Embedded Watchdogs: Build periodic watchdog monitoring into agents and services, outputting formatted live console cards detailing uptime, network addresses (with IP/MAC change detection), and query success/fail statistics."
- Obecny watchdog nie monitoruje zmian adresów IP/MAC

**Skutek:**

- Brak informacji o zmianach sieciowych

**Sugerowana naprawa:**
Dodać nową funkcję watchdog:
```python
import netifaces

async def monitor_network():
    """Monitoruje zmiany adresów IP i MAC."""
    previous_ips = {}
    previous_macs = {}
    
    while True:
        try:
            current_ips = {}
            current_macs = {}
            
            for interface in netifaces.interfaces():
                # IP addresses
                addrs = netifaces.ifaddresses(interface)
                if netifaces.AF_INET in addrs:
                    current_ips[interface] = [a['addr'] for a in addrs[netifaces.AF_INET]]
                
                # MAC addresses
                if netifaces.AF_LINK in addrs:
                    current_macs[interface] = [a['addr'] for a in addrs[netifaces.AF_LINK]]
            
            # Wykryj zmiany
            for interface, ips in current_ips.items():
                if interface not in previous_ips or set(ips) != set(previous_ips.get(interface, [])):
                    logger.info(f"[NETWORK] Zmiana IP na interfejsie {interface}: {previous_ips.get(interface, [])} -> {ips}")
            
            for interface, macs in current_macs.items():
                if interface not in previous_macs or set(macs) != set(previous_macs.get(interface, [])):
                    logger.info(f"[NETWORK] Zmiana MAC na interfejsie {interface}: {previous_macs.get(interface, [])} -> {macs}")
            
            previous_ips = current_ips
            previous_macs = current_macs
            
        except Exception as e:
            logger.warning(f"Błąd monitorowania sieci: {e}")
        
        await asyncio.sleep(60)
```

def list_files(project: str, limit: int = 100, offset: int = 0):
    """Zwraca listę plików z paginacją."""
    proj_path = Path(SCANS_DIR) / project
    if not proj_path.exists():
        raise HTTPException(status_code=404, detail="Projekt nie znaleziony")
    
    all_files = []
    for root, _, filenames in os.walk(proj_path):
        for f in filenames:
            rel_path = Path(os.path.join(root, f)).relative_to(proj_path)
            all_files.append(str(rel_path).replace("\\", "/"))
    
    return {
        "files": all_files[offset:offset+limit],
        "total": len(all_files),
        "offset": offset,
        "limit": limit
    }

## 🔄 Pełna akwizycja modelu 3D — Przewodnik

### ❓Czy system obsługuje pełny obrót 360°?

**Tak, ale wymaga odpowiedniej konfiguracji parametrów skanowania.**

### 📐 Konfiguracja dla pełnego skanu 3D

System scAnt używa **3 osi**:
- **Oś X** (0-45mm): Ramię kamery — ruch poziomy (lewo-prawo)
- **Oś Y** (0-360°): Stół obrotowy — **obrót modelu** ⭐
- **Oś Z** (-250 do -80mm): Wózek ostrości — ruch pionowy (góra-dół)

**Domyślne ustawienia** (z `api_server.py` i `Scanner_Controller.py`):
- X: min=0, max=45, step=5 → **10 pozycji**
- Y: min=0, max=160, step=8 → **20 pozycji** (tylko 160°, nie pełne 360°)
- Z: min=-250, max=-80, step=50 → **4 pozycje**

**→ Domyślnie NIE jest to pełny obrót 360°!**

---

### 🎯 Jak wykonać pełny skan 360° modelu

#### 1. **Pełny obrót (360°) w płaszczyźnie Y**

Aby zeskować cały model dookoła:

```bash
# Przez API (cURL)
curl -X POST http://localhost:8000/scan/start \
  -H "Content-Type: application/json" \
  -d '{
    "project_name": "full_rotation_scan",
    "x_min": 0, "x_max": 45, "x_step": 5,
    "y_min": 0, "y_max": 360, "y_step": 1,  # ← Pełne 360° z krokiem 1°
    "z_min": -250, "z_max": -80, "z_step": 50
  }'
```

```bash
# Przez CLI
python scant_cli.py scan \
  --project full_rotation_scan \
  --x-min 0 --x-max 45 --x-step 5 \
  --y-min 0 --y-max 360 --y-step 1 \
  --z-min -250 --z-max -80 --z-step 50 \
  --wait
```

**⚠️ UWAGA:**
- `y_max=360` i `y_step=1` wygeneruje **360 zdjęć** tylko dla osi Y
- Z osiami X (10 pozycji) i Z (4 pozycje): **360 × 10 × 4 = 14,400 zdjęć**
- Czas skanowania: ~14,400 × (czas na zdjęcie + ruch) ≈ **kilka godzin**
- Rozmiar danych: ~14,400 × 10MB = **~144 GB** (dla RAW)

#### 2. **Rekomendowane ustawienia dla pełnego modelu**

| Jakość | Y Range | Y Step | X Range | X Step | Z Range | Z Step | Liczba zdjęć | Czas (est.) | Rozmiar |
|--------|---------|-------|---------|-------|---------|-------|--------------|-------------|---------|
| **Niska** | 0-360° | 10° | 0-45mm | 10mm | -250 to -80mm | 100mm | 36 × 5 × 2 = **360** | ~30-60 min | ~3-6 GB |
| **Średnia** | 0-360° | 5° | 0-45mm | 5mm | -250 to -80mm | 50mm | 72 × 10 × 4 = **2,880** | ~2-4 godz. | ~20-40 GB |
| **Wysoka** | 0-360° | 2° | 0-45mm | 5mm | -250 to -80mm | 25mm | 180 × 10 × 7 = **12,600** | ~8-12 godz. | ~100-150 GB |
| **Maksymalna** | 0-360° | 1° | 0-45mm | 2mm | -250 to -80mm | 10mm | 360 × 23 × 17 = **142,320** | ~2-3 dni | ~1-1.5 TB |

#### 3. **Optymalizacja dla różnych typów modeli**

##### 🟢 **Małe obiekty (np. monet, biżuteria)**
- **Y:** 0-360°, step=5° (72 pozycje)
- **X:** 0-20mm, step=2mm (11 pozycji) — mniejszy zakres
- **Z:** -100 to -50mm, step=10mm (6 pozycji) — płaski obiekt
- **Liczba zdjęć:** 72 × 11 × 6 = **4,752**

##### 🟡 **Średnie obiekty (np. rzeźba 20cm)**
- **Y:** 0-360°, step=3° (120 pozycji)
- **X:** 0-45mm, step=5mm (10 pozycji)
- **Z:** -200 to -80mm, step=30mm (5 pozycji)
- **Liczba zdjęć:** 120 × 10 × 5 = **6,000**

##### 🔴 **Duże obiekty (np. busta 50cm)**
- **Y:** 0-360°, step=2° (180 pozycji)
- **X:** 0-45mm, step=3mm (16 pozycji)
- **Z:** -250 to -50mm, step=20mm (11 pozycji)
- **Liczba zdjęć:** 180 × 16 × 11 = **31,680**

##### 🟣 **Cylindryczne obiekty (np. butelka, waza)**

- **X:** 0-20mm, step=10mm (3 pozycje) — mniejszy zakres poziomy
- **Z:** -250 to -80mm, step=50mm (4 pozycje)
- **Liczba zdjęć:** 180 × 3 × 4 = **2,160**

### ⚠️ **Typowe problemy i rozwiązania**

#### Problem: Stół obrotowy nie obraca się pełne 360°
**Przyczyna:** Ograniczenie mechaniczne lub konfiguracja Klippera
**Rozwiązanie:**
- Sprawdź konfigurację Klippera (`config/skr_pico_klipper.cfg`)
- Ustaw `position_max` dla osi Y na 360
- Sprawdź fizyczne ograniczenia mechaniczne

#### Problem: Zbyt długo trwa skanowanie
**Przyczyna:** Zbyt mały krok (y_step=1 generuje 360 zdjęć tylko na Y)
**Rozwiązanie:**
- Zwiększ y_step do 5° lub 10°
- Ogranicz zakres X i Z
- Użyj mniejszej rozdzielczości kamery

#### Problem: Zbyt duże zużycie dysku
**Przyczyna:** Zdjęcia RAW zajmują ~10MB każde
**Rozwiązanie:**
- Przetwarzaj zdjęcia w batchach (np. po 1000 zdjęć)
- Użyj kompresji tymczasowych plików
- Przenoś pliki do chmury w trakcie skanowania

#### Problem: Nierównomierne oświetlenie
**Przyczyna:** Źródło światła z jednej strony
**Rozwiązanie:**
- Użyj pierścieniowego światła (ring light)
- Obracaj źródło światła razem z modelem
- Zrób kilka zdjęć z różnych kątów oświetlenia

#### Problem: Zamazane zdjęcia
**Przyczyna:** Ruch kamery podczas naświetlania
**Rozwiązanie:**
- Zwiększ `settle_time` w `wait_for_moves()` (scripts/Scanner_Controller.py:71)
- Użyj krótszego czasu naświetlania
- Sprawdź stabilność mechaniczna

## 🆕 Metoda 5: Nowy endpoint API (bezpośredni upload)

### 1. Dodaj metodę uploadu do S3

```python
# W scripts/Scanner_Controller.py
import subprocess
import tempfile

def upload_to_s3(self, file_path, s3_path):
    """Wgrywa plik do S3 używając rclone."""
    try:
        result = subprocess.run(
            ["rclone", "copy", file_path, s3_path],
            capture_output=True,
            text=True,
            timeout=60
        )
        result.check_returncode()
        return True
    except subprocess.CalledProcessError as e:
        logging.error(f"Błąd uploadu do S3: {e.stderr}")
        return False
```

### 2. Zmodyfikuj `capture_image` aby przesyłać do S3

```python
# W scripts/camera_controller.py
def capture_image(self, img_name: str):

            s3_path = f"s3_scant:scant-backups/{img_name}"
            if upload_to_s3(img_name, s3_path):
                logging.info(f"Zdjęcie przesłane do S3: {s3_path}")
                # Opcjonalnie: usuń lokalny plik
                # os.remove(img_name)
        
    except Exception as e:
        logging.error(f"Błąd podczas robienia zdjęcia: {e}")
```

1. Przejdź do **IAM** → **Users** → **Add user**
2. Nazwa użytkownika: `scant-s3-user`
3. Wybierz: **Programmatic access**
4. **Permissions:** `AmazonS3FullAccess` (lub bardziej restrykcyjna polityka)
5. Zapisz **Access Key ID** i **Secret Access Key**

### 2. Restrykcyjna polityka IAM (rekomendowana)

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "s3:PutObject",
                "s3:GetObject",
                "s3:ListBucket",
                "s3:DeleteObject"
            ],
            "Resource": [
                "arn:aws:s3:::my-bucket",
                "arn:aws:s3:::my-bucket/*"
            ]
        }
    ]
}
```

### 3. Konfiguracja rclone z IAM

```bash
rclone config create s3_scant s3 \
  provider=AWS \
  env_auth=false \
  access_key_id=YOUR_ACCESS_KEY \
  secret_access_key=YOUR_SECRET_KEY \
  region=us-east-1 \
  acl=private \
  storage_class=STANDARD
```

### 4. Test połączenia

```bash
# Utwórz testowy plik
echo "test" > /tmp/test.txt

# Wgraj do S3
rclone copy /tmp/test.txt s3_scant:my-bucket/test/

# Sprawdź czy plik jest w S3
rclone ls s3_scant:my-bucket/test/

# Usuń testowy plik
rclone delete s3_scant:my-bucket/test/test.txt
rm /tmp/test.txt
```

## 💡 Porady dla S3

### 1. **Organizacja bucketu**

```
s3://my-bucket/
├── scant/
│   ├── projects/
│   │   ├── project_001/
│   │   │   ├── RAW/
│   │   │   │   ├── x_p00000_y_p00000_step_n00000_.tif
│   │   │   │   └── ...
│   │   │   └── stacked/
│   │   │       └── stacked_image.tif
│   │   └── project_002/
│   │       └── ...
│   └── backups/
│       └── 2026-06-30/
│           └── full_backup.tar.gz
└── other-data/
```

### 2. **Klasy storage (optymalizacja kosztów)**

| Klasa | Opis | Koszt | Czas dostępu | Użycie |
|-------|------|-------|--------------|--------|
| STANDARD | Standardowa | $$$ | Milisekundy | Aktywne projekty |
| STANDARD_IA | Dostęp nieczęsty | $ | Milisekundy | Archiwum (30+ dni) |
| ONEZONE_IA | Jedna strefa | $ | Milisekundy | Archiwum (30+ dni) |
| INTELLIGENT_TIERING | Automatyczne zarządzanie | $$ | Milisekundy | Zmienne zużycie |
| GLACIER | Archiwum | $ | Minuty-godziny | Długoterminowe archiwum |
| GLACIER_DEEP_ARCHIVE | Głębinowe archiwum | $ | 12 godzin | Bardzo rzadko używane |

**Rekomendacja:**
- Aktywne projekty: **STANDARD**
- Projekty >30 dni: **STANDARD_IA**
- Archiwum >90 dni: **GLACIER**

**Konfiguracja w rclone:**
```bash
rclone config edit s3_scant
# Zmień storage_class na np. STANDARD_IA
```

### 3. **Lifecycle Policy (automatyczne przenoszenie)**

**W AWS Console → S3 → Bucket → Management → Lifecycle Rules:**

```
Rule 1: Przenoszenie do IA po 30 dniach
- Transition to: STANDARD_IA
- Days after creation: 30

Rule 2: Przenoszenie do Glacier po 90 dniach
- Transition to: GLACIER
- Days after creation: 90

Rule 3: Usunięcie po 365 dniach (opcjonalnie)
- Permanently delete
- Days after creation: 365
```

### 4. **Wersjonowanie (Versioning)**

Włącz wersjonowanie bucketu, aby chronić przed przypadkowym usunięciem:

**W AWS Console → S3 → Bucket → Properties → Versioning:**
- Enable versioning: **Yes**

### 5. **Monitorowanie kosztów**

```bash
# Sprawdź rozmiar bucketu
rclone size s3_scant:my-bucket

# AWS CLI: Koszty S3
aws cloudwatch get-metric-statistics \
  --namespace AWS/S3 \
  --metric-name BucketSizeBytes \
  --dimensions Name=BucketName,Value=my-bucket \
  --start-time 2026-06-01T00:00:00Z \
  --end-time 2026-06-30T23:59:59Z \
  --period 86400 \
  --statistics Average
```

## ⚠️ Typowe problemy z S3 i rozwiązania

### Problem: Błąd "Access Denied"
**Przyczyna:** Nieprawidłowe poświadczenia lub uprawnienia IAM
**Rozwiązanie:**
- Sprawdź poprawność `access_key_id` i `secret_access_key`
- Upewnij się, że IAM User ma uprawnienia S3
- Użyj `rclone config show` aby sprawdzić konfigurację

```bash
# Sprawdź konfigurację rclone
rclone config show

# Test połączenia z debugowaniem
rclone -vv ls s3_scant:my-bucket
```

---

### Problem: Wolne przesyłanie
**Przyczyna:** Mała liczba równoległych transferów lub niska przepustowość
**Rozwiązanie:**
- Zwiększ liczbę transferów: `--transfers 16` (domyślnie 4)
- Użyj opcji `--fast-list` dla dużych bucketów
- Rozważ użycie CDN (CloudFront) dla częstego dostępu

```bash
# Szybsze przesyłanie
rclone copy /local/path s3_scant:bucket/ \
  --transfers 16 \
  --fast-list \
  --progress
```

---

### Problem: Błąd "The AWS Access Key Id needs to be specified"
**Przyczyna:** Brak poświadczeń w konfiguracji rclone
**Rozwiązanie:**
```bash
# Ponownie skonfiguruj remote
rclone config edit s3_scant
# Upewnij się, że access_key_id i secret_access_key są ustawione
```

---

### Problem: Błąd "region is required"
**Przyczyna:** Brak ustawionego regionu
**Rozwiązanie:**
```bash
rclone config edit s3_scant
# Ustaw region, np. us-east-1, eu-west-1 itp.
```

---

### Problem: Duplikowanie plików w S3
**Przyczyna:** Wielokrotne uruchamianie `rclone copy` z tymi samymi plikami
**Rozwiązanie:**
- Użyj `rclone sync` zamiast `rclone copy` (synchronizuje, nie duplikuje)
- Użyj opcji `--checksum` aby porównywać sumy kontrolne
- Sprawdź, czy nie masz wielu skryptów uruchomionych równolegle

```bash
# Synchronizacja (nie kopiuje duplikatów)
rclone sync /local/path s3_scant:bucket/ --checksum
```

---

### Problem: Błąd "No such file or directory" w S3
**Przyczyna:** Ścieżka w S3 nie istnieje (S3 nie ma klasycznej struktury katalogów)
**Rozwiązanie:**
- rclone **automatycznie tworzy** nieistniejące "katalogi" w S3
- Upewnij się, że syntaktyka ścieżki jest poprawna: `bucket:nazwa/katalog/plik`
- Sprawdź, czy bucket istnieje: `rclone lsd s3_scant:`

---

### Problem: Wysokie koszty S3
**Przyczyna:** Zbyt częste operacje (PUT, GET, LIST) lub nieoptymalna klasa storage
**Rozwiązanie:**
- Użyj `rclone size` aby sprawdzić rozmiar bucketu
- Skonfiguruj Lifecycle Policy (przenoszenie do tańszych klas)
- Ogranicz liczbę operacji LIST (użyj `--fast-list`)
- Rozważ użycie S3 Intelligent-Tiering

```bash
# Sprawdź rozmiar bucketu
rclone size s3_scant:my-bucket

# Wyświetl koszty (AWS CLI)
aws ce get-cost-and-usage \
  --time-period Start=2026-06-01,End=2026-06-30 \
  --granularity MONTHLY \
  --metrics "UnblendedCost" \
  --group-by Dimension=SERVICE
```

---

### Problem: Błąd "Connection reset by peer" / Timeout
**Przyczyna:** Nietrwałe połączenie internetowe lub zbyt długi timeout
**Rozwiązanie:**
- Zwiększ timeout: `--timeout 1h` (domyślnie 5 minut)
- Użyj opcji `--retries` i `--retries-sleep`
- Sprawdź połączenie internetowe

```bash
# Dłuższy timeout i retry
rclone copy /local/path s3_scant:bucket/ \
  --timeout 1h \
  --retries 10 \
  --retries-sleep 30s
```

---

### Problem: Pliki nie są widoczne w AWS Console
**Przyczyna:** Opóźnienie w propagacji (S3 jest eventually consistent)
**Rozwiązanie:**
- Odczekaj kilka minut
- Odśwież konsolę AWS
- Użyj `rclone ls` aby sprawdzić czy pliki są w S3

```bash
# Sprawdź zawartość bucketu
rclone ls s3_scant:my-bucket/projects/
```

## 🎯 **Rekomendacja finalna**

Dla **Raspberry Pi** z systemem scAnt, **najlepszym rozwiązaniem** jest:

### **🥇 Metoda 2: Automatyczna synchronizacja (skrypt + cron)**

**Dlaczego?**
- ✅ Prosta konfiguracja
- ✅ Nie obciąża Raspberry Pi (synchronizacja w tle)
- ✅ Pełna kontrola nad procesem
- ✅ Bez modyfikacji kodu
- ✅ Działa z każdą chmurą (S3, Google Drive, Dropbox itp.)

**Kroki:**
1. Skonfiguruj rclone na Raspberry Pi
2. Utwórz skrypt synchronizacji
3. Ustaw cron na automatyczne uruchamianie
4. Gotowe! 🎉

**Przykładowy workflow:**
```bash
# 1. Zeskanuuj model
python scant_cli.py scan --project my_model --wait

# 2. Automatycznie zsynchronizuj do S3 (przez cron)
# Skrypt sync_to_s3.sh uruchamiany co 30 minut

# 3. (Opcjonalnie) Przetwórz w chmurze
python scant_cli.py runpod --project my_model --remote-path s3_scant:my-bucket/scant
```
